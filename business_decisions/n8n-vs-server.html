n8n vs. Server Code for the Document Pipeline

Recommended: move to server Pending final go-ahead
The short answer

Build it in the server.

This pipeline is code-heavy and file-heavy — exactly where n8n's strengths matter least and its limits hurt most. The one thing n8n gave for free (background execution) is replaced cleanly by pg-boss on the Postgres you already run.

The framing

The core insight

The question isn't "is n8n good?" — it's whether n8n fits this pipeline.

n8n's superpower is wiring together APIs with little or no code — "when a row hits this table, send a Slack message, then create a Trello card." But this pipeline is, at almost every step, custom code: unzip a file, surgically edit XML, chunk text, call an AI, merge results by index, re-zip, upload. We're already writing Code nodes for all of it. So we pay n8n's costs (binary-handling quirks, browser-based editing, dependency allow-lists, another service to run) without collecting its main benefit. That mismatch is the heart of this decision.

The why

Reasoning, dimension by dimension

File handling — the biggest factor

Code

A .docx is a zip, and n8n's binary/base64 handling between nodes is clumsy; you end up building workarounds. In Node, fflate unzips and re-zips a docx in a few lines, working with normal Buffers. The thing fighting us today simply disappears.

AI calls & the split/merge logic

Code

Chunk-and-merge (split paragraphs into batches, send each to the AI, merge back by paragraph index) is naturally a loop over an array — a for loop or a Promise pool with a concurrency cap, ~30 lines. In n8n it becomes the "Loop + sub-workflow + re-attach metadata" dance, its most error-prone part.

Maintainability & AI-assisted dev

Code

The goal is to have code written for you, not to troubleshoot nodes by hand. n8n workflows are large JSON blobs that don't diff well, can't be unit-tested easily, and can't be read or edited by an assistant. In the repo, the assistant writes it, git tracks it, and each transform is testable in isolation.

Deployment & security

Code

You already run an Express server. Adding the pipeline there means one thing to deploy and one place the Supabase service-role key lives. n8n is a whole extra service to host, patch, and back up — and a second copy of sensitive credentials.

Long-running jobs — n8n's genuine edge

n8n

A 50-page document with AI calls takes minutes, which must never run inside a web request. n8n decouples this automatically. Code-only needs a background queue — but pg-boss runs on your existing Postgres, so the gap is real but small and well-solved.

Observability

n8n

n8n's visual execution history (inspect each step, replay failures) is genuinely nice for debugging flow. In code this is replicated with logging, an error tracker, and a processing_stage column — more setup, but standard and arguably more powerful once in place.

The call

Recommendation

Move the pipeline into the server code. For this workload the reasoning is one-sided: code-heavy and file-heavy (n8n's weak spots), it benefits enormously from AI-assisted development (impossible inside n8n), and your server and database absorb the work with no extra infrastructure.

Keep n8n for what it's great at: lightweight integrations where you don't want to write or deploy code — e.g. "email me when a project sits unprocessed for an hour." Just not as the engine for document processing.

Honest caveat: there's real, working effort already in the n8n flow, and rebuilding has a cost. But the migration is mostly moving logic you've already figured out into cleaner files — not redesigning it — and every future change gets easier, not harder.
